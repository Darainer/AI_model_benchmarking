⏱ Only the green-outlined stage is inside the latency timer. The load / read / preprocess work runs in the loop but is not charged to latency — which is why latency and end-to-end FPS diverge (see §2).

2 · How to read the numbers

Read latency, not FPS, for model speed.

The timer wraps only model.infer(), so avg / p95 latency are clean, decode-independent inference numbers. The FPS column is frames ÷ wall-clock and the wall clock includes reading + JPEG-decoding each 1920×1080 image (cv2.imread, CPU) every iteration — so FPS here is image-read-bound, not inference-bound. The inf/s column (1000 ÷ avg-latency) is the honest model throughput.

Legend. avg/p95 ms = per-inference latency (timed) · inf/s = 1000/avg (pure inference) · FPS = end-to-end incl. image read (read-bound here) · effBW = weight bytes streamed ÷ latency (lower-bound DRAM demand, §4) · GPU MHz = avg GPU clock (devfreq) · Mem peak footprint · Pwr CPU+GPU+CV rail.

What the results say

  • CNN classifiers are fast and cheap — squeezenet ~173 inf/s @ 1.6 W, mobilenet ~125 inf/s. Real-time with headroom.
  • Transformers dominate cost — segformer/rf_detr ~15–17 inf/s; superpoint+lightglue 284 ms (~3.5/s) at ~6.9 W. They also hold the GPU at 67–82% and ramp it to full clock; the classifiers don't.
  • The classifiers ran at the GPU's minimum 306 MHz (of 1020). The dynamic governor never saw enough sustained load to ramp them, so their latency is pessimistic — locking clocks (jetson_clocks) would speed them up. The heavy models pulled the clock to 700–1020 MHz on their own.
  • Memory footprint is tight — 6.3–7.1 GB of 7.6 GB used per model; several can't co-reside (see §4).

Note it read 20.4 GB/s on an earlier run — the ceiling itself moves with the EMC clock / power state, which is why power mode must be pinned (below).

Finding: at batch = 1, none of these models are weight-bandwidth-bound. Weight streaming uses ≤8% of the measured ceiling — they are compute- and latency-bound instead (low GPU clock on the light models, kernel-launch overhead, the CPU image read). Bigger model ⇒ more weight bytes ⇒ higher effective BW (resnet50’s 51 MB pulls 3.4 GB/s vs squeezenet’s 2.5 MB at 0.4) — exactly the volume-÷-time relationship — but all well under the wall.

Caveat (important): this is a lower bound — it counts weights only. The segmentation/detection models move large activation tensors (e.g. 512×512 feature maps) that dwarf their weights, so their true DRAM traffic is far above the 0.1–0.8 GB/s shown. Measuring that needs the hardware memory-controller counters, which on Orin/JP6 are root-only (no EMC in tegrastats, no actmon sysfs, debugfs unmounted in the container). The clean way to get true live EMC bandwidth is jetson-stats (its root service exposes EMC; a non-root client reads it) — a planned follow-up.

③ Pin the power mode before quoting anything

The measured BW ceiling moved 20→43 GB/s between runs and the classifiers sat at the 306 MHz GPU floor — both are governor/power-state artefacts. For stable, comparable numbers run sudo nvpmodel -m 0 && sudo jetson_clocks (max clocks, locked) and record the mode alongside results. Unpinned clocks change these figures more than most model choices do.

5 · Scaling up: Orin NX + DLA, and the AGX question

The Orin Nano has GPU only — no DLA. Orin NX and AGX Orin add 2× NVDLA 2.0 fixed-function accelerators that run a subset of layers (mostly CNN ops) at low power, with GPU fallback for the rest.

How these workloads scale on an NX

  • Offload CNNs to DLA, keep the GPU for transformers. Pin squeezenet/mobilenet/resnet50/most-of-unet to DLACore 0/1; leave segformer/rf_detr/lightglue on the GPU. The two engine types then run concurrently — raising aggregate pipeline throughput, not just single-model speed.
  • Perf/watt is the DLA win, not peak speed — a single CNN can be marginally slower on one DLA than the GPU, but at lower power and it frees the GPU.
  • More bandwidth helps the heavy models (NX ~1.5×, AGX ~3× this Nano). Segmentation/detection activation traffic is where bandwidth actually bites — exactly the part the §4 lower bound doesn’t capture.
  • Optical flow has its own engine — cuda_optflow_mono can target the Orin OFA, off-loading the GPU entirely.

Would an AGX run accurately predict DLA performance on an NX?

For architecture/bring-up: yes. For absolute numbers: no.

Transfers (AGX → NX): same NVDLA 2.0 IP, so which layers compile to DLA vs fall back to GPU, the fallback size, INT8 accuracy, and the relative model ranking are all representative. Use AGX to validate the DLA path and decide what to offload.

Doesn’t transfer: absolute latency/throughput. AGX DLA runs at a higher clock and—decisively—has ~2× the memory bandwidth of NX. DLA on these CNNs is often bandwidth-bound, so AGX DLA over-estimates NX DLA, often a lot.

Bottom line: AGX proves the mapping and ranks models; measure on a real NX (at the target nvpmodel) for numbers you quote. AGX→NX clock+bandwidth scaling is a rough first guess only — and fix the power mode either way.

To actually test DLA in this harness

Today: ONNX Runtime TensorRT EP (FP16, GPU). DLA needs engines built for it, e.g. trtexec --onnx=m.onnx --useDLACore=0 --int8 --allowGPUFallback --saveEngine=m.dla.trt, loaded through tensorrt_backend.py. Engines are device/clock-specific — build and measure on the NX/AGX itself.
