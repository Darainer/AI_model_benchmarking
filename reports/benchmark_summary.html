⏱ The green-outlined stage is the only one inside the latency timer. The preceding load/decode/preprocess work runs in the loop but is not charged to latency — which is exactly why latency and end-to-end FPS can diverge (see §3).

Why this run uses a folder of images, not the video

Hardware video decode (nvv4l2decoder) works perfectly on the host but currently fails inside the container on this box (S_EXT_CTRLS for CUDA_GPU_ID failed — a nvidia-container-toolkit 1.16.2 issue, image-independent; see open_issues.md). Software decode works but caps throughput at the CPU decode rate (~25 FPS), masking true model speed. To isolate inference cleanly, we decoded 100 consecutive frames once on the host (HW decode) with scripts/extract_frames.sh and benchmark against that image set — no live video decode inside the timed loop.

2 · How to read the numbers

Read latency, not FPS, for model speed.

The timer wraps only model.infer(), so avg / p95 latency are clean, decode-independent inference numbers. The FPS column, however, is frames ÷ wall-clock, and the wall clock includes reading + JPEG-decoding each 1920×1080 image (cv2.imread, on CPU) every iteration. So the FPS column here is image-read-bound, not inference-bound — the same kind of confound as software video decode, just moved upstream. The giveaway is GPU utilisation: squeezenet shows only 11% GPU, i.e. the GPU is idle waiting on CPU JPEG decode. The “inference/s” column below is derived as 1000 ÷ avg-latency and is the honest throughput of the model itself.

Metric legend. avg/p95 ms = per-inference latency (timed). inf/s = 1000/avg = pure-inference throughput. FPS = end-to-end incl. image read (read-bound here). GPU% / Mem / Pwr = tegrastats averages over the run (Pwr = CPU+GPU+CV rail).

What the results say

  • The CNN classifiers are fast and cheap. squeezenet ~169 inf/s @ 1.6 W, mobilenet ~99 inf/s — these are real-time-with-headroom on a Nano.
  • Transformers dominate the cost. segformer, rf_detr and especially superpoint+lightglue (466 ms) are an order of magnitude slower and draw 2.7–4.6 W. They also have the highest GPU% — they keep the GPU busy, the classifiers don't.
  • This run was read-bound, not compute-bound. Low GPU% on the light models means the 1080p JPEG decode in the loop is the limiter for end-to-end FPS. Pre-loading frames to RAM (a “prebuffer” mode) would make the FPS column track inf/s.
  • Memory is the quiet constraint. 5.9–7.0 GB used of 7.6 GB — several of these models would not co-reside; the Nano is near its memory ceiling running them one at a time.

4 · Scaling up: Orin NX + DLA, and the AGX question

The Orin Nano has GPU only — no DLA. The Orin NX and AGX Orin add 2× NVDLA 2.0 deep-learning accelerators: fixed-function INT8/FP16 engines that run a subset of layers (mostly CNN ops — conv, pooling, element-wise) at low power, with automatic GPU fallback for anything unsupported.

How these workloads would scale on an NX

  • Offload the CNNs to DLA, keep the GPU for transformers. Pin the DLA-friendly models (squeezenet, mobilenet, resnet50, most of unet) to DLACore 0/1 and leave the poorly-mapped transformers (segformer, rf_detr, lightglue) on the GPU. The two engine types then run concurrently, raising aggregate pipeline throughput rather than just single-model speed.
  • Perf/watt, not peak speed, is the DLA win. A single CNN is often a touch slower on one DLA than on the GPU, but at much lower power — and it frees the GPU. For an always-on multi-model perception stack that’s the right trade.
  • Optical flow has its own engine. cuda_optflow_mono can target the Orin OFA (Optical Flow Accelerator), another way to take load off the GPU.
  • More bandwidth + bigger GPU helps everything. Even GPU-only, NX (~1.5× BW) and AGX (~3× BW, 2× GPU) cut the transformer latencies materially, since several of these layers are bandwidth-bound (note the modest 20.4 GB/s d2d we measured on the Nano).

Would an AGX run accurately predict DLA performance on an NX?

For architecture/bring-up: yes. For absolute numbers: no.

What transfers (AGX → NX): AGX Orin and Orin NX use the same NVDLA 2.0 IP. So everything structural is representative — which layers compile to DLA vs fall back to GPU, the size of that fallback, INT8 quantisation accuracy, supported/unsupported ops, and the relative ranking of which models suit DLA. An AGX run is an excellent, faster way to validate the DLA path and decide what to offload.

What does NOT transfer: absolute latency/throughput. The AGX DLA runs at a higher clock and—decisively—has ~2× the memory bandwidth of the NX. DLA on these CNNs is frequently bandwidth-bound, so AGX DLA numbers will over-estimate NX DLA throughput, often substantially. You cannot read an FPS off AGX and expect it on NX.

Bottom line: use AGX to prove the DLA mapping and rank models; measure on an actual NX (at the target nvpmodel power mode) for numbers you quote. A clock-and-bandwidth scaling estimate from AGX→NX is a rough first guess only. Always fix the power mode — an unmatched nvpmodel changes results more than the DLA-vs-GPU choice does.

To actually test DLA in this harness

Today the pipeline uses the ONNX Runtime TensorRT EP (FP16, GPU). DLA requires TensorRT engines built for it, e.g. trtexec --onnx=m.onnx --useDLACore=0 --int8 --allowGPUFallback --saveEngine=m.dla.trt, loaded through the repo’s native tensorrt_backend.py. Build per-target (the engine is device/clock specific) and re-measure on the NX/AGX itself.
